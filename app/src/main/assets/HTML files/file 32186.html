Yes, backing up from X and restoring to Y is a normal operation. We often call this "reinitializing the slave."

This does interrupt replication. There's no reliable way to restore the data at the same time as letting the slave continue to apply changes, because the changes the slave is processing are not in sync with the snapshot of data represented by the backup. You could overwrite changed data, or miss changes, and this would make the slave totally out of sync.

So you have to stop replication on the slave while you restore.

Here are the steps for a typical slave reinitialization:

  1. mysqldump from the master, with the --master-data option so the dump includes the binary log position that was current at the moment of the dump.
  2. Stop replication on the slave.
  3. Restore the dump on the slave.
  4. Use CHANGE MASTER to alter what binary log coordinates the slave starts at. Use the coordinates that were saved in the dump.
  5. Start slave.

Yes, there's an option for mysqldump --no-data so the output only includes the CREATE TABLE and other DDL, but no INSERT statements with data. Then you can import that to a separate database on the same server. And you're right, by default DDL statements are added to the binary log, so any replication slaves will automatically run the same statements.

You can even do the export & import in just two steps like this:

$ mysqladmin create newdatabase
$ mysqldump --no-data olddatabase | mysql newdatabase
