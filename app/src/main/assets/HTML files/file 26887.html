We had a similar need in our project, and we ended up defining a fake entity in our model that didn't not have a corresponding table in the physical database. We used this entity type to return data from the stored procedure.

Here's roughly what we did. Our stored procedure returned a collection of a type DistributionFiles that could be defined as a complex type. Instead we defined it as an entity type. It resulted in the following model changes:

  1. Storage model: 

    <EntityType Name="DISTRIBUTION_FILES">
  <Key>
        ... keys
  </Key>
    ... properties
</EntityType>

  2. Conceptual model:

    <EntitySet Name="DistributionFiles" EntityType="Model.DistributionFiles" />

  3. Function definition:

    <FunctionImport Name="GetDistributionFiles" EntitySet="DistributionFiles" ReturnType="Collection(Model.DistributionFiles)">

  4. Mapping:

    <EntityTypeMapping TypeName="Model.DistributionFiles">
  <MappingFragment StoreEntitySet="DISTRIBUTION_FILES">
    ... field mapping
  </MappingFragment>
</EntityTypeMapping>

Then I could use DistributionFiles entity as there was a corresponding table. There only place it was really used was a stored procedure GetDistributionFiles that returned a collection of this entities.
