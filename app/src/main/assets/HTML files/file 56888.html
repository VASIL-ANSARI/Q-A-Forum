There is no way to tell if a device's clock is correct with 100% certainty. That said, you can use AJAX or WebSockets to try to ask a trusted server for a datetime stamp and compare that with what the device reports. At minimum, there will be a margin of error largely involving network latency.

Anyway, something practical:

<script type="text/javascript">
  function myCallback(json) {
    var internalNow = new Date();
    var externalNow = new Date(json.dateString);
    var msg = '';
    if(
         internalNow.getUTCFullYear() == externalNow.getUTCFullYear()
      && internalNow.getUTCMonth() == externalNow.getUTCMonth()
      && internalNow.getUTCDate() == externalNow.getUTCDate()
    ) {
      msg = 'Date matches.';
    }
    else {
      msg = 'Date DOES NOT match.';
    }
    alert(msg);
  }
</script>
<script type="text/javascript" src="http://www.timeapi.org/utc/now.json?callback=myCallback"></script>

This relies on www.timeapi.org which doesn't support HTTPS. I'm not aware of a similar service that does. You might be better off setting up your own time service. It also doesn't check for normal, minor fluctuations so there's likely to be issues around midnight UTC.
