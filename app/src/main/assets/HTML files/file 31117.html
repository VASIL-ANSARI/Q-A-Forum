You have a asked a very broad question with lots of opportunity for subjective answers, for which I had to vote to close it. There are lots of good parts to your question which is why I try to provide some answer.

  1. There is no preferred way. Do you want to install git as a dependency on your production/staging/web machines? If you want to, deploying with git is perfectly acceptable, if not it is just as easy to use built in rsync utility and interact with it using fabric. There are lots of scripts online showing how to set up fabric using either way. There are probably a number of other ways too but rsync or git pull seems to be the most popular ways to deploy.

  2. Fabric lets you define roles, and makes it easy to deploy to multiple servers at one time.

  3. Special cases require special functions. You can define these in your fab file. You could create the logic to deploy to your general web servers in a function then have a seperate function to build and deploy your fragmented server.

  4. Amazon provides a couple of autoscaling tools, I have never used any but I am sure that lots of people have, and lots of people have blogged about their experiences.
