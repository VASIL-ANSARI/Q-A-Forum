There is nothing wrong with using 2 different SDKs.

SDK A here is using the non-recommended method gcm.register(senderID). One of the reasons why Google recommends using the Instance Id API is because, this API provides a listener that listens for any token refreshs and requests that your app refreshes its tokens. This will ensure your app always has the latest token.

Now with SDK A, you are pushing with an old registration token, which will work, but eventually will stop working when GCM removes the old token from their server. With SDK B, you have the latest token.

Once you send a push notification, you will get in your response the latest registration token known as the canonical id. Your server side should update the old registraton token with this canonical id.

Why you get a MismatchSenderId error:

A registration token is tied to a certain group of senders. When a client app registers for GCM, it must specify which senders are allowed to send messages. You should use one of those sender IDs when sending messages to the client app. If you switch to a different sender, the existing registration tokens won't work.

I suppose, SDK A and SDK B has its own sender id, which explains why that returned token is not working for SDK A.

Workaround:-

  • For SDK A, try registering that device again when you will get the latest token which you could use to push.
  • Better still, implement the Instance Id API
