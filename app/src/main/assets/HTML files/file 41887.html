Structure

The idea of using multiple inheritance is perfectly valid here:

  • A Camera is-a node, and at the same time is-a plugin object which implements the CameraInterface.
  • The difficulty that you experience with accessor and bidirectional link from Camera to CameraInterface (to syncrhonize at least the lifetime) and from CameraInterface to Camera (to allow plugin to manipulate the real object) confirms that inheritance is structurally more adapted here than composition.

Virtual inheritance needs to be assessed carefully:

  • at first sight, it seems not necessary: your example seems to accredit the hypotheses that only the final graphic elements inherit from plugin hierarchy, i.e. the note and the plugin inheritance hierarchies are not interleaved.
  • it could however be required if the reality is more complex, and escpecially if all graphic elements could inherit from plugin elements at any levels. In this case without virtuality, you'd have several instances of plugin in the same object, which doesn't seem to make sense.

Virtual inheritance would make it slightly more complex, as you have alwys to initailise explicitely the common virtual bases.

Performance:

With multiple inheritance, you'll have several vtables for accessing virtual functions (one for each inheritance hierarchy). However, for a given pointer type, the implementation will only access a single vtable with a single offset. Thus performance will not be more degraded than single inheritance with virtual functions.

In addition you can compare with the single inheritance alternative that you have explored. You'll notice that navigation between objects and their adpaters will need a lot of pointer dereferencing, which might balance out the vtable indirection (and only if you would'nt make use of virtual functions).
