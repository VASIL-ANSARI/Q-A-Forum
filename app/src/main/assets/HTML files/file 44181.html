There are a few things involved here that it's useful to know.

First, .select2("val", xyz) is equivalent to .val(xyz).trigger("change"). The reason why a change event is triggered is for two reasons

  • Older versions of Select2 used to trigger the change event, so we gave an easy-upgrade path.
  • Other components on the page should be listening to the change event to know when the value of the <select> changes, so it makes sense to trigger it.

Second, Select2 can only know if the underlying value of the <select> changes if you trigger the change event.

So when you combine these two issues together, it starts to explain why Select2 doesn't act as you would expect when you set the new value within a change event.

You can get around the cyclic change events by using the select2:select event to determine when a new selection is made, instead of the change event.

https://jsfiddle.net/nmeoosLk/8/

This is locking you into using Select2-specific events, but it's your best alternative if you don't want to always check if the value changed. I personally would recommend not triggering the change event if the value didn't change, as the small amount of code will better help other components understand what is happening on the page.
