All of these regular expression patterns that you can supply to the code generator are patterns that match schemas, tables, or columns either by their unqualified name (e.g. version) or by their fully qualified name (e.g. public.users.version). You cannot use partially qualified names like users.version.

To answer your concrete question, the following are examples that will work:

<!-- Will match version columns in all tables -->
<recordVersionFields>version</recordVersionFields>

<!-- Will match version columns in all user tables. 
     Notice that schema names are matched by .* -->
<recordVersionFields>.*\.user\.version</recordVersionFields>

<!-- Will match version columns only in that particular public.user table -->
<recordVersionFields>public\.user\.version</recordVersionFields>

This will generate the relevant meta information into your UserRecord for optimistic locking to be enabled on UserRecord.store(), update(), and delete() calls.

Side note: Your compilation error

In your question, you referred to a compilation error that seems to happen because you're matching all version columns in all tables:

<!-- Will match version columns in all tables -->
<recordVersionFields>version</recordVersionFields>

This will also match "wrong" version columns, e.g. in the schema_version table, which is of type VARCHAR not INTEGER. Perhaps, you should be more explicit with your regular expression to do exact matching.

Side note: Case sensitivity

All of these regular expressions are case sensitive. If you're using PostgreSQL, then the following is probably wrong:

<inputSchema>PUBLIC</inputSchema>

Instead, write any of these:

<!-- use lower-case in PostgreSQL by default -->
<inputSchema>public</inputSchema>

<!-- use case-insensitive regular expressions to "stay safe" --> 
<inputSchema>(?i:PUBLIC)</inputSchema>
